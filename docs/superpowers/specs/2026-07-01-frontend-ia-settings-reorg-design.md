# 前端信息架构与配置中心重组设计

状态：设计稿，待确认后实施。
日期：2026-07-01。

## 背景

当前前端主导航中，`维护诊断` 同时包含统计、工作流、定时任务、STRM、115 网盘和配置终端。`SettingsTab` 内部又把 115 Cookie、Emby、夸克、飞牛、MoviePilot、ANI-RSS、Twilight、HDHive、TMDB、Pansou、Telegram、代理、更新、归档、热榜追更、安全、资源过滤器和实时日志放在同一组设置页内。

这造成几个具体问题：

- 夸克位于 `第三方集成`，但它实际是网盘账号和转存目标，与 115 属于同一类。
- 115 Cookie 位于 `基础连接`，115 扫码登录位于 `115 网盘管理`，115 默认转存目录和离线默认目录位于 `115 网盘管理`，归档监听/输出目录位于 `归档与追更`。同一账号域被拆散，用户难以判断入口。
- `第三方集成` 同时混合网盘、媒体库、自动化系统、搜索源、元数据源和论坛账号。
- `实时操作日志` 内存在 `触发归档扫描` 等业务操作，日志页承担了非日志职责。
- `配置与终端` 页面名过宽，无法表达账号、系统、诊断、日志、更新、安全等多个边界。

## 目标

- 建立面向用户操作心智的一级导航和配置中心子 tab。
- 将“账号/连接/默认目标”与“业务执行工作台”分离。
- 将“配置型、执行型、观察型”操作分离，减少误操作和入口歧义。
- 为后续增加阿里云盘、百度网盘、WebDAV、更多搜索源和更多媒体服务预留稳定位置。
- 尽量复用现有组件、API 和状态，不在第一阶段引入后端模型变更。
- 第一阶段优先做前端信息架构重排，不改业务行为。

## 非目标

- 不重做整体视觉设计。
- 不改变 115、夸克、归档、订阅、MoviePilot、ANI-RSS、Telegram 的后端 API 语义。
- 不把 115 文件浏览、离线下载、分享转存从工作台移入设置页。
- 不把归档监听/输出目录简单归入“网盘账号配置”；它们属于归档策略，只是引用 115 目录。
- 不在本轮实现多网盘统一文件浏览抽象；只为未来扩展预留命名和位置。
- 不新增复杂路由，除非实施阶段确认拆出独立日志页更划算。

## 设计原则

### 1. 按用户要完成的事分组

用户进入页面时通常有四类意图：

- 配账号：填 Cookie、扫码登录、填 API Key、设置默认目录。
- 做任务：浏览网盘、转存分享、添加离线下载、跑归档、启停工作流。
- 看状态：看任务统计、日志、健康状态、风控状态。
- 调策略：配置订阅优先级、归档规则、资源质量过滤、自动同步周期。

导航和 tab 应优先匹配这些意图，而不是匹配后端接口文件名。

### 2. 配置型、执行型、观察型分离

配置型：

- 凭据、地址、开关、周期、默认目录、规则。

执行型：

- 立即转存、添加离线下载、立即同步、手动签到、重建索引、清理队列。

观察型：

- 运行统计、日志、健康检查结果、账号状态、风控状态。

同一个页面可以有轻量快捷操作，但主职责必须单一。日志页只做观察和日志维护，不做业务触发。

### 3. 按对象域归类

`115`、`夸克`、未来 `阿里云盘` 属于网盘账号域。

`Emby`、`飞牛`、`Twilight` 属于媒体服务域。

`TMDB`、`Pansou`、`HDHive`、`TG 索引` 属于资源发现与元数据域，但 HDHive 同时有账号和签到行为，需要作为资源源配置卡，而不是泛“第三方集成”。

`MoviePilot`、`ANI-RSS` 属于外部自动化/订阅执行系统。

`归档`、`热榜追更`、`影人关注` 属于自动化策略。

### 4. 默认目录按用途命名

现有 115 至少有三类目录：

- 分享转存默认目录：`/pan115/default-folder`，用于分享转存。
- 离线下载默认目录：`/pan115/offline/default-folder`，用于离线任务保存。
- 归档监听/输出目录：`archive_watch_cid` / `archive_output_cid`，用于归档扫描策略。

它们不能都叫“默认目录”。必须在 UI 中显式标注用途。

## 子 Tab 详细设计

### 1. 网盘集成

定位：管理网盘账号、登录状态、默认转存目标和连通检测。

包含：

- 115 账号
  - Cookie 原始字符串。
  - 扫码登录。
  - 登录状态、用户信息、风控摘要。
  - 115 API 会话检测。
- 115 默认目标
  - 分享转存默认目录。
  - 离线下载默认目录。
  - 快捷说明：归档监听/输出目录在 `订阅与归档` 中配置。
- 夸克账号
  - Cookie。
  - 默认转存目录。
  - 连通检测。
- 未来扩展占位
  - 阿里云盘。
  - 百度网盘。
  - WebDAV。

不包含：

- 115 文件浏览。
- 115 离线任务列表。
- 115 分享链接解析与转存执行。
- 归档监听目录和输出目录的主配置。

操作规则：

- `保存网盘配置` 只保存当前网盘相关字段。
- `测试连接` 可以在保存前临时提交 Cookie 后检测，沿用现有行为，但文案必须说明会更新当前会话。
- 扫码登录成功后应刷新 115 Cookie 状态和用户信息。
- 默认目录必须按用途分为 `分享转存默认目录` 和 `离线下载默认目录`。

现有来源：

- `pan115Api.getCookieInfo/updateCookie/checkCookie`
- `pan115Api.listQrLoginApps/startQrLogin/checkQrLogin/cancelQrLogin/getQrImage`
- `pan115Api.getDefaultFolder/setDefaultFolder`
- `pan115Api.getOfflineDefaultFolder/setOfflineDefaultFolder`
- `quarkApi.getCookieInfo/updateCookie/checkConnectivity`
- `quarkApi.getDefaultFolder/setDefaultFolder`

## 网盘工作台边界

当前 `Pan115FilesTab` 应演进为 `网盘工作台`。

第一阶段仍只支持 115，但文案和结构为多网盘预留。

保留：

- 115 Cookie 状态摘要。
- 用户信息。
- 离线配额。
- 风控状态。
- 文件浏览。
- 文件搜索。
- 新建文件夹。
- 重命名、删除、复制、移动。
- 复制下载链接。
- 离线下载任务新增、重试、删除、清空已完成。
- 分享链接解析。
- 分享链接单项/批量/全量转存。

调整：

- 页面标题从 `115 网盘管理` 改为 `网盘工作台`。
- 当前账号状态卡可以保留 `重新扫码登录` 快捷入口，但文案应提示“完整账号配置在配置中心 > 网盘集成”。
- 分享转存默认文件夹和离线默认文件夹可以保留“使用当前目录设置”的快捷操作，因为它和文件浏览上下文强相关。
- 默认目录主配置仍在 `配置中心 > 网盘集成`，工作台只是快捷设置。

不包含：

- Cookie 原始串输入。
- 完整账号配置表单。
- 归档监听/输出目录配置。

## STRM 管理边界

`StrmTab` 当前同时展示 STRM 配置、生成、诊断和归档入口。

建议边界：

- STRM 配置、立即生成、STRM 诊断继续保留在 `STRM 管理`。
- 自动归档摘要可以保留，但 `归档参数` 的主配置入口应指向 `配置中心 > 订阅与归档`。
- `立即扫描归档` 可以保留一个快捷按钮，因为 STRM 用户常需要在生成前后触发归档，但按钮文案必须明确“这是归档任务，不是 STRM 生成”。

## 工作流与定时任务边界

`工作流`：

- 面向自动化规则：事件、cron、动作编排、手动执行。

`定时任务`：

- 面向底层调度任务：启停、编辑、删除、运行。

配置中心中的周期设置保存后可以创建或更新 scheduler task，但用户查看调度状态仍进入 `定时任务`。

## 操作流设计

### 首次配置网盘

```text
配置中心 -> 网盘集成 -> 115 账号 -> 扫码登录或填 Cookie -> 检测连接
配置中心 -> 网盘集成 -> 设置分享转存默认目录 / 离线下载默认目录
网盘工作台 -> 浏览文件 / 添加离线任务 / 分享转存
```

### 配置夸克转存

```text
配置中心 -> 网盘集成 -> 夸克账号 -> 填 Cookie -> 设置默认转存目录 -> 检测连接
详情页 / 搜索结果 -> 夸克资源 -> 转存到夸克默认目录
```

## 组件拆分建议

第一阶段可以不拆文件，只重排 `SettingsTab` 渲染结构。但为了后续维护，建议实施时逐步拆组件：

```text
SettingsTab.tsx
- SettingsSidebar
- settings/CloudDrivesSettings.tsx
- settings/MediaServicesSettings.tsx
- settings/ResourceMetadataSettings.tsx
- settings/TelegramSettings.tsx
- settings/SubscriptionArchiveSettings.tsx
- settings/SystemNetworkSettings.tsx
- settings/SecurityPreferencesSettings.tsx
- settings/OperationLogsPanel.tsx
```

状态管理建议：

- 第一阶段继续使用 `SettingsTab` 顶层 state，降低改动风险。
- 拆子组件时通过 props 传入字段和 handler。
- 第二阶段再考虑 `useSettingsRuntimeForm` 这类 hook，避免一次性大重构。

保存策略：

- 保留顶部 `保存当前全部配置`，但每个对象域也应保留局部保存按钮。
- 局部保存按钮优先保存本域字段，减少用户误以为保存了其他域。
- 高风险执行按钮不应触发全局保存。

不要使用模糊文案：

- `刷新`

- `执行`

默认目录文案必须带用途：

- `分享转存默认目录`

## UI 行为要求

- 左侧子 tab 在桌面端可 sticky，移动端横向滚动。
- 子 tab 数量增加后，每项必须有清晰 label 和短描述。
- 状态不能只靠颜色表达，必须有文字标签。
- 高风险动作需要确认：
  - 清理归档队列。
  - 全量重构 TG 索引。
  - 清理旧日志。
  - 删除/清空类操作。
- 超过 300ms 的动作显示 loading。
- 日志和诊断结果就近展示，不只写入 toast。
- 任何操作按钮 hover 不应导致布局跳动。

## 风险与处理

### 风险 1：SettingsTab 已经过大，直接重排容易引入状态遗漏

处理：

- 第一阶段不改 state 结构，只移动渲染块。
- 先完成 tab schema 和迁移映射，再按块移动。
- 每移动一块就跑 build。

### 风险 2：同一字段在多个位置有快捷入口

处理：

- 保留一个主配置位置。
- 快捷入口文案必须标明“快捷设置”或“完整配置位置”。
- 避免两处表单同时编辑同一字段但状态不同步。

### 风险 3：用户习惯旧入口

处理：

- 第一阶段可以保留旧 tab 的空壳或提示，但不建议长期保留。
- 在新 tab 描述中写清对象域，例如 `网盘账号、登录与默认目录`。

### 风险 4：资源质量偏好归属不唯一

处理：

- 推荐放 `订阅与归档`，因为它直接影响订阅抓取和转存选择。
- 如果用户更倾向把它看作个人偏好，可放 `安全与偏好`，但不要同时出现两份。
